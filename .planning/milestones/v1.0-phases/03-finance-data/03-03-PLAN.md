PocketBase transactions collection fields:
- user: relation (required) — always set to pb.authStore.record?.id
- account: relation to accounts (required)
- type: select ["income", "expense", "transfer_in", "transfer_out"] (required)
- amount_centavos: number, onlyInt, min:1 (required)
- date: date (required) — user-specified date, not the auto-created timestamp
- category: relation to categories (optional — not required for transfers)
- note: text, max 300 (optional)
- exchange_rate_stored: number, onlyInt (optional — only for USD account transactions)
- transfer_pair_id: text, max 30 (optional — both legs of a transfer set this to each other's ID)

dolarapi.com endpoint:
  GET https://dolarapi.com/api/dolares — returns array of rate objects
  Key fields per object: { casa: string, venta: number | null, compra: number | null }
  casa values we use: "blue", "oficial", "tarjeta"
  Use Next.js fetch cache: next: { revalidate: 300 } (5 minutes)
  ALWAYS call from a Server Action. NEVER fetch from the browser/client.

Transfer pattern (TWO linked records — see research for full implementation):
  1. Create transfer_out record on fromAccount with transfer_pair_id: '' initially
  2. Create transfer_in record on toAccount with transfer_pair_id: outRecord.id
  3. Update outRecord to set transfer_pair_id: inRecord.id (cross-reference)
  On delete: always look up transfer_pair_id and delete both records

Pitfall: dolarapi.com unavailability must NOT block form submission.
  Rate fetch is best-effort. Manual input must ALWAYS be available and pre-filled with fetched value when available.

Pitfall: archived accounts must not appear in the account selector on TransactionForm.
  Filter: archived = false in account list query.

Pitfall: category field is optional for transfer_in/transfer_out types.
  The UI should hide the category selector when type is "transfer".

Create `lauos/src/lib/actions/transactions.ts` with five exported Server Actions:

**fetchDolarRatesAction** — fetches dolarapi.com and returns blue/oficial/tarjeta venta rates:
```typescript
export async function fetchDolarRatesAction(): Promise<{
  rates?: { blue: number; oficial: number; tarjeta: number }
  error?: string
}>
```
Use `fetch('https://dolarapi.com/api/dolares', { next: { revalidate: 300 } })`. Extract venta values for casa "blue", "oficial", "tarjeta". If fetch fails or rate is null, return `{ error: 'No se pudo obtener la cotización. Ingresala manualmente.' }`. This is the ONLY place dolarapi.com is called — never in client code.

**createTransactionAction** — creates a single income or expense transaction:
```typescript
export async function createTransactionAction(data: {
  accountId: string
  type: 'income' | 'expense'
  amountCentavos: number
  date: string          // ISO date string "YYYY-MM-DD"
  categoryId: string
  note?: string
  exchangeRateStored?: number  // only if account currency is USD
}): Promise<{ id?: string; error?: string }>
```
- Check `!pb.authStore.isValid` → return error
- Set `user: pb.authStore.record?.id` explicitly
- `exchange_rate_stored` included only when provided (not 0)
- `revalidatePath('/finance')` after success

**createTransferAction** — creates two linked transaction records atomically:
```typescript
export async function createTransferAction(data: {
  fromAccountId: string
  toAccountId: string
  amountCentavos: number
  date: string
  note?: string
  userId: string   // caller passes pb.authStore.record?.id (validated before call)
}): Promise<{ error?: string }>
```
Implementation (from research pattern):
1. Create transfer_out record with transfer_pair_id: ''
2. Create transfer_in record with transfer_pair_id: outRecord.id
3. Update outRecord to set transfer_pair_id: inRecord.id
4. If step 2 or 3 fails, attempt to delete outRecord and return error
5. `revalidatePath('/finance')` after success

Note: userId is passed explicitly because the action needs it in two create calls. Validate inside: if `!pb.authStore.isValid` return error.

**updateTransactionAction** — updates amount, date, category, note on an existing income/expense transaction. Does NOT allow type changes or transfer edits (too complex — show "Transfer edits not supported" error if type is transfer_in/transfer_out):
```typescript
export async function updateTransactionAction(data: {
  id: string
  amountCentavos?: number
  date?: string
  categoryId?: string
  note?: string
  exchangeRateStored?: number
}): Promise<{ error?: string }>
```

**deleteTransactionAction** — deletes one transaction record. If `transfer_pair_id` is non-empty, fetches and deletes the paired record first (ignore 404 errors on the pair — it may already be gone):
```typescript
export async function deleteTransactionAction(id: string): Promise<{ error?: string }>
```
Pattern:
```typescript

  try { await pb.collection('transactions').delete(record.transfer_pair_id) } catch {}
}
await pb.collection('transactions').delete(id)
revalidatePath('/finance')
```
